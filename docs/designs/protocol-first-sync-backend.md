## Vision

### 10x Check
AgentFleet doesn't just support multiple storage backends. It becomes the universal standard for serverless agent coordination. The protocol spec is published RFC-style. A conformance test suite ships alongside it. Backend authors publish npm packages. Fleet-wide observability is built in from day one. The protocol is so clean you can debug coordination issues by reading JSON files in a directory.

### Platonic Ideal
You run `agentfleet init --backend local-folder --path /shared` and in 30 seconds, two machines are coordinating agents. The protocol is human-readable, the fleet status is always visible, adding a new backend is implementing the SyncBackend interface (~10 methods) and publishing an npm package. Post-v1: the web dashboard auto-discovers the backend and adapts (v1 stays OneDrive-only for the dashboard).

## Implementation Approach
Approach B: Protocol-First Architecture (chosen over Minimal Viable and Adapter-Per-Provider). The delta from Minimal Viable is small with CC but the architectural difference is massive. Protocol-first is what makes AgentFleet a platform, not a tool.

**PRODUCT MODEL CHANGE (identified by outside voice):** This plan changes AgentFleet from a broadcast fan-out system (every agent runs every task) to a single-claim distributed task queue (each task is claimed and executed by exactly one agent). This is an intentional product evolution, not just a backend refactor. It warrants a major version bump to v3.0.0.

### Implementation Sequencing
Phase 1 (ship as v3.0.0): Protocol spec, SyncBackend interface, local-folder backend, backend registry, CLI --backend flag, init wizard, conformance test suite, poll-claim loop (replaces event-driven watcher)
Phase 2 (fast-follow v3.1.0): Fleet agent discovery, fleet-status command, doctor command, dry-run mode, task priority field

**Dropped from Phase 1 (per outside voice):** OneDrive CLI backend refactor. The CLI has no OneDrive backend — it uses local filesystem symlinks to the local OneDrive sync folder. A real OneDrive SyncBackend (Graph API, auth, token storage) is Phase 3+ work. Existing OneDrive symlink behavior is preserved in v3.0.0 as a legacy compatibility mode.

Phase 1 is the platform. Phase 2 is the observability layer. Both ship in this plan's scope, but Phase 1 must be solid before starting Phase 2.

**Phase 1 gate:** Conformance test suite passes for local-folder backend, `agentfleet init` can set up local-folder backend, poll-claim loop correctly executes single-claim semantics, and all i18n keys have parity.

### Runner Architecture Change
The current event-driven model (Chokidar fires on new file → agent executes immediately) is replaced with a poll-claim loop:
1. Scan `tasks/` directory periodically (configurable interval, default 10s)
2. Sort pending tasks by priority (higher first, then FIFO)
3. For each unclaimed task: attempt claim via convergence-then-tiebreak protocol
4. Execute claimed tasks via AgentExecutor (unchanged)
5. Watcher remains as optional optimization: triggers an immediate scan cycle instead of waiting for poll interval

This model naturally enables: single-claim semantics, priority ordering, dry-run (scan + sort + show, skip claim + execute), and fleet-status (scan + show state).

### SyncBackend Interface (full specification)
The design doc (chenxizhang-main-design-20260411-141603.md) specifies the complete interface. Key methods:
- `name: string` — backend identifier
- `initialize(config: BackendConfig): Promise<void>` — set up connection, create fleet directories (`tasks/`, `claims/`, `heartbeats/`, `results/`, `fleet/`) if they don't exist
- `shutdown(): Promise<void>` — clean up resources, close watchers
- `writeFile(path, content): Promise<void>` — write string content; creates parent directories if needed; overwrites if file exists
- `readFile(path): Promise<string>` — read file contents; throws if file doesn't exist
- `listFiles(directory): Promise<string[]>` — list file names in a directory; returns empty array if directory is empty
- `deleteFile(path): Promise<void>` — delete a file; no-op if file doesn't exist
- `watchDirectory(path, callback): Promise<Watcher>` — watch for file add/change/unlink events
- `fileExists(path): Promise<boolean>` — check file existence
- `createExclusive(path, content): Promise<boolean>` — atomic create-if-not-exists; returns false if file already exists; never overwrites
- `getRecommendedConvergenceWindow(): number` — returns convergence window in ms (local-folder: 5000ms, onedrive: 60000ms, representing ~2x expected sync latency)

Implementation per backend:
- **local-folder:** `createExclusive` uses `fs.open` with `O_CREAT | O_EXCL` flags. `watchDirectory` uses `fs.watch` with polling fallback.
- **onedrive:** `createExclusive` is best-effort (upload with `@microsoft.graph.conflictBehavior: fail`). On OneDrive, `createExclusive` may return `true` for multiple agents in a race. The convergence-then-tiebreak protocol handles this case. The convergence window MUST be long enough for OneDrive sync to propagate all claim files (default: 60s).
- **NFS caveat:** `O_EXCL` is not reliable on NFS v2/v3. Document as known limitation. NFSv4 supports it.

### Task Claim Protocol (full lifecycle)
1. Agent discovers unclaimed task in `tasks/` (status: `pending`)
2. Agent writes claim file to `claims/{task-id}/{agent-id}` via `createExclusive`
3. Agent waits `convergence_window` (local: 5s min, cloud: 60s min, configurable)
4. Agent re-reads all files in `claims/{task-id}/`
5. Agent with lexicographically lowest ID wins. Comparison uses JavaScript's `<` operator on strings. Agent IDs MUST be ASCII-only (hostnames and PIDs are always ASCII).
6. Losers delete their claim files and skip the task
7. Winner updates task status to `claimed`, then `running`
8. Winner writes heartbeat to `heartbeats/{task-id}` every 30s (configurable)
9. On completion: winner writes result to `results/{task-id}.json`, updates task status to `completed` or `failed`, deletes heartbeat and claim
10. On crash recovery: any agent observing a heartbeat older than 3x interval marks the task status as `timeout` (observable state), then deletes stale claim + heartbeat, and task reverts to `pending` for re-claiming

**Convergence-then-tiebreak explained:** On eventually-consistent storage, two agents may both successfully write claim files because sync hasn't propagated yet. The convergence window is a deliberate pause that lets sync catch up. After the pause, all agents see all claims, and the tiebreak (lowest agent ID) is deterministic. This converges without coordination.

### Protocol Versioning
Every task envelope includes `protocol_version: 1`. Agents MUST reject tasks with a protocol_version they don't support. Forward-compatibility rule: agents SHOULD ignore unknown fields in task envelopes (allows adding optional fields without version bump). A version bump is required only for breaking changes to required fields or claim/heartbeat semantics.

### Error Handling & Retry Semantics
- **Transient errors** (network timeout, disk full): retry with exponential backoff (3 attempts, 1s/2s/4s). After all retries fail, log error and skip the task.
- **Permanent errors** (file not found, permission denied): fail immediately, log with full context.
- **All write operations must be idempotent.** Use write-to-temp-then-rename where the backend supports it.
- **Partial writes must never leave corrupt state.** Temp files are cleaned up on failure.

### Fleet Agent Discovery Schema
`fleet/{agent-id}.json`:
```json
{
  "agentId": "hostname-pid",
  "hostname": "machine-name",
  "pid": 12345,
  "startedAt": "2026-04-12T01:00:00Z",
  "lastHeartbeat": "2026-04-12T01:05:00Z",
  "status": "idle|busy",
  "currentTask": null,
  "backend": "local-folder",
  "protocolVersion": 1
}
```
Staleness threshold: 3x the fleet heartbeat interval (default: fleet heartbeat every 60s, stale after 180s). Stale agents are shown as "unreachable" in fleet status.

### Migration Path (v2 → v3)
This is a major version bump (v2.x → v3.0.0) because the product model changes from broadcast fan-out to single-claim queue.
1. Clean break: v2.x = broadcast (every agent runs every task), v3.x = single-claim (one agent per task)
2. Mixed fleets (v2 + v3 agents sharing a directory) are NOT supported. v2 agents ignore claims and would duplicate-execute tasks.
3. Existing users run `agentfleet init` to migrate their config to the v3 format
4. OneDrive symlink behavior is preserved in v3 as legacy compatibility mode (not a SyncBackend, just the existing path detection)
5. Protocol-v1 directories (`claims/`, `heartbeats/`, `results/`, `fleet/`) are created alongside existing `tasks/` and `output/`

### Open Questions Resolved
1. **Heartbeat interval:** 30s default, configurable per-agent via config. Not per-backend (heartbeat is agent behavior, not backend behavior).
2. **Config format precedence:** env vars > CLI flags > config.json > defaults. Same pattern as most CLI tools.
3. **Convergence window scope:** Per-backend via `getRecommendedConvergenceWindow()`, overridable per-fleet in config.json `convergenceWindowMs` field.

### Task Priority Semantics
Agents SHOULD claim higher-priority tasks first (priority field: integer, default 0, higher = more urgent). Priority sorting is agent-side behavior, not a backend responsibility. Agents that don't support priority treat all tasks equally. The protocol does not enforce priority ordering.

### Conformance Test Risk
Concurrent claim simulation tests use a mock clock (fake timers) to avoid real convergence delays. Real-timing tests against actual backends are manual or run in a dedicated CI stage. This avoids flaky tests from timing dependencies.

### Security Model (v1 threat assessment)
All agents in a fleet are trusted. A malicious agent can: read all tasks, forge results, claim and abandon tasks indefinitely (DOS), delete other agents' claims. This is acceptable for v1 on trusted internal networks and personal file sync. Documented as explicit limitation. Future versions may add agent identity verification and signed task envelopes.

### Chokidar + Cloud Sync Caveat
Local-folder backend over a cloud-synced directory (Dropbox folder, OneDrive local folder) is a "best-effort" configuration. Cloud sync clients write partial files that trigger spurious events. Mitigation: `stabilityThreshold` tuning in the watcher config. For production cloud use, prefer the native cloud backend adapter.

### Conformance Test Strategy
- Tests run against an in-memory mock backend AND the local-folder backend
- Cloud backends (OneDrive) get conformance-tested manually or in a separate authenticated CI stage
- Test categories: basic CRUD, createExclusive atomicity, claim protocol correctness, heartbeat lifecycle, concurrent agent simulation

## Key Decisions Made
1. `createExclusive` is essential in SyncBackend interface (prior learning, confidence 9/10)
2. Cloud backends use convergence-then-tiebreak as primary correctness mechanism; createExclusive is optimization
3. Protocol v1 uses trusted-agent security model (documented limitation)
4. Web dashboard stays OneDrive-only for v1 (explicit gap, acceptable, post-v1 goal)
5. Chokidar v5 is kept as optional watcher optimization; core model is poll-claim loop
6. Implementation ships in two phases: platform first, observability second
7. Task priority is agent-side sorting behavior; agents SHOULD sort by priority but the protocol does not enforce it
8. Major version bump to v3.0.0 (product model change from broadcast to single-claim)
9. OneDrive CLI backend dropped from Phase 1 (CLI has no OneDrive backend, just symlinks)
10. Mixed v2+v3 fleets are NOT supported (clean break)
11. Runner architecture changes from event-driven watcher to poll-claim loop
12. ProtocolEngine class encapsulates all claim/heartbeat/discovery logic
13. `getFileModifiedTime` added to SyncBackend interface for efficient heartbeat checks
14. Path validation on all task IDs and file paths (traversal protection)
15. All 4 error gaps from Section 2 fixed (PermissionError, CorruptWriteError, WatcherLimitError, HeartbeatWriteError)
16. OneDrive golden path integration test added for deployment safety

## Engineering Review Findings

Review conducted 2026-04-12 by /plan-eng-review. Score: 8/10.

### Architecture Decisions (Section 1, 5 issues resolved)
1. **Result format:** Protocol result (`results/{task-id}.json`) goes to shared directory via SyncBackend. Debug logs (stdout/stderr) stay local at `~/.agentfleet/logs/{task-id}/`. ResultWriter splits into two concerns.
2. **Mandatory init:** `agentfleet run` without prior `agentfleet init` shows error. No auto-bootstrap. Clean break.
3. **Watcher as scan trigger:** Chokidar detects file change → calls `scanner.triggerImmediateScan()` → full poll-claim cycle. No direct execution from watcher events.
4. **Status in task file:** Only the winning claimer writes status updates. Status is informational (for fleet-status/doctor), not a locking mechanism. Use write-to-temp-then-rename for atomicity.
5. **processedIds removed:** Protocol state is single source of truth. Drop `processed.json`.

### Code Quality Decisions (Section 2, 4 issues resolved)
1. **Command injection fix:** Rewrite `spawnAgent` to split command into argv array, pass prompt as separate argument. `spawn(command, [...args, prompt], { shell: false })`. No shell interpretation of prompts.
2. **Path traversal protection:** `ProtocolEngine.validateId()` rejects IDs containing `..`, `/`, `\`, null bytes, or non-alphanumeric-dash-underscore characters. Single validation point.
3. **New config type:** v3 `AgentFleetConfig` replaces OneDrive fields with `backend` (string) + `backendConfig` (object). `agentfleet init` handles migration from v2 format.
4. **Typed error hierarchy:** `src/backends/errors.ts` with `BackendError` base class and `transient: boolean` property. ProtocolEngine retries transient (3 attempts, exponential backoff), fails immediately on permanent.

### Test Decisions (Section 3, 3 issues resolved)
1. **Mock backend:** `test/conformance/mock-backend.ts` implements SyncBackend as in-memory `Map<string, string>`. Conformance tests parameterized: run against mock AND local-folder.
2. **Claim race testing:** Sequential interleaved simulation. Two ProtocolEngine instances, same backend, different agent IDs. Mock timers skip convergence. Deterministic, no flakes.
3. **Full test coverage:** Every new module gets a test file. ~8-10 test files for Phase 1. Test-first convention from AGENTS.md honored.

### Performance Decisions (Section 4, 2 issues resolved)
1. **Archive completed tasks:** After writing result, ProtocolEngine moves task from `tasks/` to `archive/`. Poll scan only reads `tasks/`. Full cleanup: write archive → delete claims dir → delete heartbeat → delete from tasks.
2. **Visible convergence UX:** Progress indicator during convergence wait with countdown/spinner. Log outcome (won/lost, claimant count).

### Outside Voice Findings (Eng Review, 6 critical gaps resolved)
1. **Claim-age timeout:** If task is `claimed` but has no heartbeat after 2x convergence window, any agent can scavenge it back to `pending`. Covers winner-crashes-before-running case.
2. **At-least-once delivery guarantee:** Stated explicitly in PROTOCOL.md. At-most-once under normal operation, at-least-once under network partition. Task handlers SHOULD be idempotent.
3. **Agent ID uniqueness:** Format changed to `{hostname}-{pid}-{random4}`. Random suffix generated at init time, stored in config. Prevents collisions in containerized environments.
4. **Rejected status for malformed tasks:** Any agent can transition `pending → rejected` for bad JSON, missing fields, unsupported protocol_version. Prevents infinite retry loops. State machine updated.
5. **Full archive cleanup:** Write archive → clean up claims/{task-id}/ dir → clean up heartbeats/{task-id} → delete tasks/{task-id}.json. Doctor command handles orphaned artifacts.
6. **v2 task migration:** `agentfleet init` detects v2 tasks (no protocol_version field), asks user to archive or delete them, writes VERSION file to fleet root.

### Updated State Machine
```
pending → claimed → running → completed
  ↓                        ↘ failed
  rejected                  → timeout → pending
```

### Additional Outside Voice Issues (acknowledged, not blocking Phase 1)
- **Convergence short-circuit for single-agent fleets:** If only 1 claim exists after a brief check (e.g., 1s), skip remaining convergence. Implementation detail, not protocol change.
- **Debug log rotation:** Add configurable retention (default: 100 tasks or 500MB, whichever comes first). `agentfleet doctor` reports log disk usage.
- **File integrity on partial sync:** `readFile` + JSON parse failures treated as transient errors (retry after backoff) for task/claim files. Handles partial file delivery from sync clients.
- **Clock skew:** Use `getFileModifiedTime` (filesystem time, not agent clock) for heartbeat staleness checks. Already in the interface.
- **Submit command warning:** When no recent agent presence is detected, `agentfleet submit` warns "No active agents detected. Tasks will queue until an agent starts."
